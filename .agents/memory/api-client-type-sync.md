---
name: Generated API client type sync
description: How to keep orval-generated and manually-maintained types in api-client-react in sync with the actual API
---

## Rule
`lib/api-client-react/src/generated/api.schemas.ts` is orval-generated but treated as **manually maintained** — it will NOT be regenerated automatically. When the API adds new fields, extend the relevant interfaces there directly.

Fields that have been added manually to `Product`:
- `isBestDeal`, `storeName`, `nameAr`, `hasVariants`, `viewCount` (product enrichment)
- `flashSaleEnd`, `flashSaleDiscount` (flash sale system, on PublicSettings too)

`BestSellerProduct` in `lib/api-client-react/src/best-sellers.ts` is a parallel interface used when the best-sellers endpoint returns data. It must include ALL fields that `ProductCard` reads from its `product: Product` prop. Required fields (non-optional) must match `Product` exactly — if `Product.featured: boolean` then `BestSellerProduct.featured: boolean` (not `featured?: boolean`).

`SellerApplication` interface was added manually to `api.schemas.ts` (not generated by orval) since the seller-applications endpoint wasn't in the original OpenAPI spec.

## Why
orval codegen isn't re-run in this project; the generated file is the source of truth for shared types. Type mismatches cause `tsc` failures across marketplace, mobile, and api-server simultaneously.

## How to apply
When adding a new API field: (1) add to the DB schema/route, (2) add to `api.schemas.ts` Product/relevant interface, (3) if used in ProductCard/BestSellers, also add to `BestSellerProduct`. Match optionality carefully.
