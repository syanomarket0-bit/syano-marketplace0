NOTE: This step implements in-process caching (no Redis required).
If Redis WAS available (detected in STEP 1.2) → implement Redis caching instead and skip this part.

STEP 4.1 — Create a generic cache service:
Create file: artifacts/api-server/src/services/cacheService.ts

Requirements:
  - Uses the existing LRU cache pattern already in searchCache.ts as reference
  - Generic typed cache: CacheService<T>
  - Constructor params: maxSize (number), defaultTtlMs (number)
  - Methods:
      get(key: string): T | undefined
      set(key: string, value: T, ttlMs?: number): void
      delete(key: string): void
      deleteByPrefix(prefix: string): void  — deletes all keys starting with prefix
      clear(): void
      size(): number
      stats(): { size: number; hits: number; misses: number; hitRate: number }
  - Internal: track hit/miss counts for monitoring
  - Thread-safe: Node.js is single-threaded, no locking needed
  - No `any` TypeScript type
  - Export as: export const cacheService = new CacheService<unknown>(1000, 5 * 60 * 1000)
  - Also export the class for typed instantiation

STEP 4.2 — Create typed cache instances:
In artifacts/api-server/src/services/cacheService.ts, also export:

  // Products list cache — invalidated on product create/update/delete
  export const productsCache = new CacheService<ProductsListResponse>(200, 60 * 1000);

  // Product detail cache — TTL 5 minutes
  export const productDetailCache = new CacheService<ProductDetailResponse>(500, 5 * 60 * 1000);

  // Categories cache — invalidated rarely, TTL 1 hour
  export const categoriesCache = new CacheService<CategoriesResponse>(10, 60 * 60 * 1000);

  // Sellers directory cache — TTL 2 minutes
  export const sellersCache = new CacheService<SellersResponse>(100, 2 * 60 * 1000);

Use the actual TypeScript response types from the existing codebase — read them before writing these.

STEP 4.3 — Apply cache to products list endpoint:
In artifacts/api-server/src/routes/products.ts — products list handler:
  
  Cache key pattern: `products:list:${page}:${pageSize}:${category}:${sort}:${minPrice}:${maxPrice}:${search}`
  
  Logic:
    1. Build cache key from all query params
    2. Check productsCache.get(cacheKey)
    3. If hit → return cached response with header X-Cache: HIT
    4. If miss → run existing DB query, store result, return with header X-Cache: MISS
  
  Cache invalidation:
    On POST /api/products (create) → productsCache.clear()
    On PUT/PATCH /api/products/:id (update) → productsCache.deleteByPrefix("products:list:")
    On DELETE /api/products/:id → productsCache.deleteByPrefix("products:list:")
  
  Do not change the response shape. Do not change existing middleware or auth checks.
