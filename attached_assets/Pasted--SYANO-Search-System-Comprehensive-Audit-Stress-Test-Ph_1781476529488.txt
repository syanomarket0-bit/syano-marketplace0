### SYSTEM OVERVIEW (what we built)
- Step 1: searchProcessor.ts — Arabic/English text cleaning, stemming, normalization, hamza unification, glued-word splitting (شاحنسريع → شاحنة سريعة), plural/singular handling
- Step 2: fts_vector column + GIN index on products table, auto-update DB trigger
- Step 3: Intent analysis backend — price intent (رخيص/cheap), quality intent (أفضل/best), category mapping (بواط → shoes/أحذية), Intent Chips on Shop page UI
- Step 4: /api/search/suggestions endpoint — returns intent/category/store/trending suggestions, debounced 200ms, parallel DB queries, query_logs table
- Step 5: Hybrid Ranking — 65% FTS + 20% exact-match bonus + 25% trigram (typo resistance)

### AUDIT AXIS 7 — Performance Benchmarks

Run these checks and report actual measured numbers:

**DB query speed:**
EXPLAIN ANALYZE SELECT * FROM products WHERE fts_vector @@ plainto_tsquery('arabic', 'موبايل');
→ Must show "Index Scan using products_fts_idx" (not Seq Scan)
→ Execution time must be < 50ms

EXPLAIN ANALYZE SELECT * FROM products WHERE name ILIKE '%موبايل%';
→ Compare: must be SLOWER than fts_vector query (proves GIN index is working)

**Suggestions API speed:**
Run 20 consecutive requests to /api/search/suggestions?q=مو&lang=ar
→ Average response time must be < 100ms
→ No request above 200ms

**Concurrent load test:**
Simulate 10 parallel requests to /api/search?q=موبايل
→ All must respond under 200ms
→ No 500 errors

**Hybrid ranking formula verification:**
Run: SELECT id, name, ts_rank(fts_vector, query) as fts_score FROM products, plainto_tsquery('arabic', 'موبايل') query ORDER BY fts_score DESC LIMIT 10;
→ Confirm scores are non-zero and results are relevance-ordered
