GOAL
────
Right now the `clicked` column in `query_logs` is never written as `true`,
so the CTR metric in the Search Analytics Panel always shows 0%.
This task makes CTR real: when a user clicks a search result, the system
writes `clicked = true` to the correct `query_logs` row.
No new DB tables. No schema changes. Only the existing `clicked` boolean
column in `query_logs` gets used.

── Change 1: Return log ID from search endpoint ──────────
In the existing GET /api/search handler, the fire-and-forget insert into
query_logs already happens. Modify it so that:
- The inserted row's `id` is captured (use Drizzle's `.returning({ id })`)
- The `id` is included in the search API response as `searchLogId: number | null`
- If the insert fails silently (fire-and-forget), searchLogId = null
- This must NOT slow down the search response — keep it non-blocking.
  Use this pattern:

  let searchLogId: number | null = null;
  db.insert(queryLogs)
    .values({ ... })
    .returning({ id: queryLogs.id })
    .then(([row]) => { searchLogId = row?.id ?? null; })
    .catch(() => {});

  Since fire-and-forget is async, the cleanest approach is to await the
  insert but with a timeout guard so it never blocks more than 50ms:
  
  const insertPromise = db.insert(queryLogs).values({...}).returning({id: queryLogs.id});
  const timeoutPromise = new Promise<null>(res => setTimeout(() => res(null), 50));
  const result = await Promise.race([insertPromise, timeoutPromise]);
  const searchLogId = Array.isArray(result) ? (result[0]?.id ?? null) : null;

  Then include searchLogId in the final JSON response alongside results.

── Change 2: New endpoint — record a click ───────────────
Add to the existing search router (same file as GET /api/search):

POST /api/search/click
Auth: optional (works for both guests and logged-in users)
Body (Zod-validated):
  {
    searchLogId: number   // the query_logs row to mark as clicked
  }

Logic:
- Validate searchLogId is a positive integer
- Update query_logs SET clicked = true WHERE id = searchLogId
- Only update if clicked is currently false (do not double-count)
- Return: { success: true } or { success: false } — nothing else
- This endpoint must respond in < 10ms — single indexed PK lookup
- If searchLogId does not exist: return { success: false }, not a 404
- No auth required — guests can also generate CTR signals

Error handling:
try {
  const { searchLogId } = ClickSchema.parse(req.body);
  await db.update(queryLogs)
    .set({ clicked: true })
    .where(and(
      eq(queryLogs.id, searchLogId),
      eq(queryLogs.clicked, false)
    ));
  res.json({ success: true });
} catch (error) {
  if (error instanceof ZodError)
    return res.status(400).json({ error: error.errors });
  console.error('[search-click]', error);
  res.status(500).json({ success: false });
}

── Add recordSearchClick function ────────────────────────
In the search-related api client file, add a plain async function
(NOT a TanStack Query hook — this is fire-and-forget, not a query):

export async function recordSearchClick(searchLogId: number): Promise<void> {
  if (!searchLogId) return;
  try {
    await fetch('/api/search/click', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ searchLogId }),
    });
  } catch {
    // silent — CTR logging must never affect UX
  }
}

const handleResultClick = (productId: number) => {
  // Fire-and-forget — never await, never block navigation
  if (searchLogId) {
    recordSearchClick(searchLogId);
  }
  // Immediately navigate — do not wait for the above

Rules:

- NEVER show loading state for this call
- NEVER show error state if it fails

  (added in Step 2) stored in component state or from the
  TanStack Query result data
- Only call recordSearchClick once per result click
  (not on hover, not on mount)
- Works for both guest and logged-in users
