over refining them. This task adds the full Discovery layer:
1. Faceted Filters — price range, rating, category, store, in-stock only

shareable and the back button works correctly.

── Filter logic to add to the Drizzle query ──────────────

Add each filter as a conditional WHERE clause.
Only apply a filter if the param is present and valid.
All filters are additive (AND logic — never OR between filters).

Price filter:
  if (minPrice !== undefined) where.push(gte(products.price, minPrice))
  if (maxPrice !== undefined) where.push(lte(products.price, maxPrice))

Rating filter:
  if (minRating !== undefined) where.push(gte(products.rating, minRating))

Category filter:
  if (categoryId !== undefined) where.push(eq(products.categoryId, categoryId))
  Also include products whose category's parent_id = categoryId
  (so filtering by "إلكترونيات" includes all sub-categories)

Store filter:
  if (storeId !== undefined) where.push(eq(products.storeId, storeId))

In-stock filter:
  if (inStock === true) where.push(gt(products.stockQuantity, 0))
  (verify actual column name in schema first)

── New fields in search response ─────────────────────────

Add to the existing response object (do not change existing fields):

  filterMeta: {
    priceRange: { min: number, max: number },  // actual min/max from
                                                // ALL products matching
                                                // the text query (before
                                                // price filter applied)
    totalUnfiltered: number,   // count before filters, after text search
    appliedFilters: {          // echo back what filters are active
      minPrice: number | null,

      categoryId: number | null,
      storeId: number | null,

The priceRange must reflect the actual data range for the current
text query — this drives the price slider bounds on the frontend.
Run this as a parallel query alongside the main results query.

── New endpoint: GET /api/search/filter-options ──────────
Add this new endpoint to the search router.
No auth required. Cached-friendly (results rarely change).

Query params (Zod-validated):
  q: string (optional — if provided, scope options to matching products)
  categoryId: number (optional)

Response shape:
{
  categories: Array<{
    id: number,
    nameAr: string,
    nameEn: string,
    slug: string,
    productCount: number    // count of products in this category
                            // matching current q if provided
  }>,
  stores: Array<{
    id: number,
    nameAr: string,
    nameEn: string,
    slug: string,
    productCount: number
  }>,
  priceRange: {
    min: number,
    max: number
  },
  processingTimeMs: number
}

Logic:
- Only return categories/stores that have at least 1 matching product
- If q is provided, scope counts to products matching that text query
- Run all sub-queries in parallel via Promise.all()
- Order categories by productCount DESC
- Order stores by productCount DESC
- This powers the filter sidebar so only relevant options show

── New endpoint: GET /api/search/related ─────────────────
Add this new endpoint to the search router.
No auth required.

Query params (Zod-validated):
  q: string (required)
  lang: 'ar' | 'en' (default 'ar')
  limit: number (default 6, max 10)

Response shape:
{
  related: Array<{
    query: string,
    count: number     // how many times this was searched
  }>,
  processingTimeMs: number
}

Logic:
- Source: query_logs table
- Find queries that share at least one normalized word with q
- Exclude q itself
- Order by count DESC
- If query_logs has < 10 entries total, fall back to returning
  top searched queries from the same category as current results
- Never return empty array — always try to return something useful

── Add useFilterOptions hook ─────────────────────────────
export function useFilterOptions(q?: string, categoryId?: number) {
  return useQuery({
    queryKey: ['search', 'filter-options', q, categoryId],
    queryFn: ({ signal }) => fetchFilterOptions({ q, categoryId }, signal),
    staleTime: 60 * 1000,   // filter options change rarely
    enabled: true,           // always fetch, even without q
  });
}

── Add useRelatedSearches hook ───────────────────────────
export function useRelatedSearches(q: string, lang: string) {
  return useQuery({
    queryKey: ['search', 'related', q, lang],
    queryFn: ({ signal }) => fetchRelatedSearches({ q, lang }, signal),
    enabled: !!q && q.length >= 2,
    staleTime: 5 * 60 * 1000,
  });
}

── URL-based filter state ────────────────────────────────
All filter values must live in the URL as query params.
Use React Router's useSearchParams for reading and writing filter state.
Never store filter values in useState only — they must be in the URL.

Reading filters from URL:
  const [searchParams, setSearchParams] = useSearchParams();
  const minPrice = Number(searchParams.get('minPrice')) || undefined;
  const maxPrice = Number(searchParams.get('maxPrice')) || undefined;
  const minRating = Number(searchParams.get('minRating')) || undefined;
  const categoryId = Number(searchParams.get('categoryId')) || undefined;
  const storeId = Number(searchParams.get('storeId')) || undefined;
  const inStock = searchParams.get('inStock') === 'true';

Updating a filter (example):
  const updateFilter = (key: string, value: string | null) => {
    const next = new URLSearchParams(searchParams);
    if (value === null || value === '') next.delete(key);
    else next.set(key, value);
    next.set('page', '1');   // reset to page 1 on filter change
    setSearchParams(next);
  };
