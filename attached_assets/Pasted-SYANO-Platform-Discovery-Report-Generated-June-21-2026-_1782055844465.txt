2. Role Inventory
customer
Default role for all new registrations
Permissions: Browse, search, cart, checkout, order placement and tracking, messaging with sellers, wishlist, AI support, submit product/seller reviews, rate couriers
Protected routes: /checkout, /orders/*, /customer/dashboard, /messages, /support, /account
Dashboard: /customer/dashboard — recent orders, account stats
Key features: Order history, real-time delivery tracking, courier rating
seller
Acquired by applying via /seller/apply (requires admin approval)
Permissions: Full product CRUD (own products only), variant management, inventory control, order status updates (own orders), store profile, analytics, customer messaging
Protected routes: All /seller/* routes
Dashboard: /seller/dashboard — revenue, follower count, active products, recent orders
Key features: Product wizard, variant builder, store trust score, earnings analytics
courier
Acquired by applying via /courier/apply (requires admin approval)
Permissions: Toggle online/offline availability, accept/decline mission offers, report pickup/delivery progress, upload delivery proof, submit safety incidents, manage wallet/payout requests
Protected routes: All /courier/* routes (also accessible to admin)
Dashboard: /courier/dashboard + /courier (workspace) — live mission map, offer flow, GPS tracking
Key features: Real-time OSRM routing, wallet with payout requests, performance metrics, mission history
admin
Bootstrapped via ROOT_ADMIN_PASSWORD env var on first API start
Permissions: Full platform control — all user management, order override, product moderation, content management (hero banners), platform settings, dispatch center, courier location monitoring, payout approval, AI support configuration
Protected routes: All /admin/* routes; can also access all /courier/* routes
Dashboard: /admin — live stats, revenue, new users, pending applications, system health alerts
Key features: Dispatch center, real-time tracking monitor, audit logs, search analytics, seller/courier application review
Public (Unauthenticated)
Permissions: Browse products, categories, stores, search, view product detail, store pages, all static/legal pages; guest wishlist (localStorage)
No login required: Home, /shop, /search, /products/*, /categories, /store/:slug, /stores, all /about/*, all /help, all legal pages

9. Search Architecture
Overview
A 13-step hybrid NLP search pipeline combining PostgreSQL FTS, semantic vector search, and a multi-level fallback chain. Implemented in artifacts/api-server/src/utils/searchProcessor.ts and artifacts/api-server/src/routes/search.ts.

NLP Pipeline (13 Steps)
Sanitization — strips injection fragments, control characters
Language detection — identifies Arabic vs. English dominant script
Arabic normalization — removes diacritics, normalizes Alef/Yaa/Hamza variants, converts Eastern digits
English normalization — lowercasing, e-commerce stemming (e.g., "shoes" → "shoe")
Syrian dialect mapping — colloquial → standard Arabic (e.g., "بواط" → "أحذية")
Intent modifier detection — detects price/quality intent signals (cheap → cheap, فخم → premium)
Stopword removal — Arabic particles and dialect fillers
Sticky token decomposition — splits concatenated Arabic compounds ("سماعةبلوتوث" → two tokens)
Cross-language bridge — injects transliterations/synonyms ("iphone" ↔ "ايفون")
FTS query construction — builds weighted tsquery (OR broad + AND precision)
Semantic embedding — encodes normalized query via embedding service (384-dim)
Parallel execution — FTS + semantic queries run simultaneously
RRF reranking — Reciprocal Rank Fusion merges and scores both result sets
FTS Setup
Column: products.fts_vector (tsvector)
Weight tiers: A = name/name_ar, B = category/subcategory, C = brand tokens, D = description (240 char truncation)
Index: GIN index on fts_vector
Sync: Postgres trigger products_fts_trig rebuilds vector on any product update
Ranking: ts_rank_cd() combined with pg_trgm trigram similarity
Semantic Embeddings
Model: paraphrase-multilingual-MiniLM-L12-v2, 384-dimensional
Storage: products.embedding column (vector(384))
Backfill: scripts/generateEmbeddings.ts
Real-time: Auto-embedded on POST product creation; updated on PATCH via generateSingleEmbedding()
Similarity: Cosine distance via pgvector <=> operator
Fallback Chain (4 Levels)
Level	Strategy	Trigger
L1	Synonym expansion	Zero results from primary
L2	Category-only broadening	Zero results from L1
L3	Trigram-only fuzzy match	Zero results from L2
L4	Global trending/popular products	Zero results from L3 (never-empty guarantee)
Embedding Service Fallback
If model.safetensors is missing or corrupt: TF-IDF + Truncated SVD (LSA) vectors — same API contract, lower quality

Caching
Type: Custom LRU cache (SearchCache)
Capacity: 500 entries
Key: MD5 hash of query parameters
TTL: 1 min (sale items), 5 min (normal), 10 min (fallback results)
Location: artifacts/api-server/src/services/searchCache.ts
10. Courier Architecture
Overview
V3.3 Courier Dispatch System. A real-time logistics engine covering discovery, offer management, mission lifecycle, GPS tracking, and financial payouts.

Mission State Machine
PENDING → SEARCHING → ASSIGNED → ACCEPTED → PICKED_UP → IN_TRANSIT → DELIVERED
                ↓                                               ↓
         NO_COURIER_FOUND                               FAILED / CANCELLED
                                                      RESCHEDULE_REQUIRED

V3.3 Dispatch Engine (missionAssignmentEngine.ts)
Parameter	Value
Rounds per mission	3
Offer window	60 seconds
Couriers per round	3
Distance algorithm	Haversine formula
Freshness filter	GPS location must be <60 seconds old
Eligibility filter	approved + ONLINE + isAcceptingDeliveries + no active mission
Failure escalation	After 3 rounds → NO_COURIER_FOUND → Dispatch Alert created
Availability States
State	Meaning	Trigger
ONLINE	Available for offers	Manual toggle by courier
OFFLINE	Not accepting offers	Manual toggle by courier
BUSY	Actively on a mission	Auto-set on mission ASSIGNED
→ ONLINE	Restored after mission	Auto-set on DELIVERED or FAILED
GPS & Tracking Pipeline (trackingService.ts)
High-frequency GPS ingestion via POST /api/courier/location
SSE stream for real-time position updates to tracking page
OSRM road routing: two-leg route (courier→seller pickup, seller→customer dropoff)
ETA calculation: segment-based with speed averaging
Tracking sessions: grouped by mission, position history stored in tracking_positions
Wallet & Payouts (courierWalletService.ts)
Component	Detail
Trigger	Mission status → DELIVERED
Courier share	~80% of delivery fee
Ledger tables	courier_wallets (materialized balance) + courier_wallet_transactions (source of truth)
Balance types	available, pending, lifetime_earnings
Payout flow	Courier requests → available → pending → Admin approve/reject
Reject effect	Amount returned to available
Safety & Incident System
Couriers can submit mission_safety_events: ROAD_HAZARD, THEFT, ACCIDENT, etc.
Events immediately trigger admin notifications via dispatch_alerts
Admins view and resolve alerts from dispatch center
Admin Dispatch Center (/admin/dispatch-center)
Live mission monitoring (all statuses)
GPS freshness indicator per courier: FRESH (<30s), WARNING (30–60s), STALE (>60s)
Manual courier assignment override
Dispatch alert inbox with resolution workflow
Delivery zone management
