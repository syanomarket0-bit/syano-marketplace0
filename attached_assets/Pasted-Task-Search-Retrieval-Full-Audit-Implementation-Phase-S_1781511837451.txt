──────────────────────────────────────────────────────────────────

  2a. Replace the current ranking formula with this calibrated version.

                         COALESCE(average_rating, 0) / 5.0 * 0.6 +

      final_score =
        (text_score     × 0.55) +
        (quality_score  × 0.20) +

IMPROVEMENT 3 — Smart Fallback Chain
──────────────────────────────────────────────────────────────────

Goal: zero results should NEVER be the final answer.
Always give the user something useful to continue their journey.

  3a. Implement a 4-level fallback chain in search.ts.
      Execute levels in order — stop at first level that returns
      results. Each level must complete in < 20ms.

      LEVEL 1 — Relaxed FTS (try immediately after zero results):
        Remove the least significant token from the query
        (shortest word first, or last word if equal length).
        Re-run the FTS query with the shortened query.
        If results > 0: return them with a flag
          fallback: { level: 1, original_query: '...', 
                      relaxed_query: '...' }

      LEVEL 2 — Trigram fuzzy match:
        Use pg_trgm similarity search on product titles only.
        Threshold: similarity > 0.25 (low threshold = more results).
        Limit: 10 results maximum.
        If results > 0: return them with flag
          fallback: { level: 2, match_type: 'fuzzy' }

      LEVEL 3 — Category inference fallback:
        Take the original query. Check if any token matches
        a category name (case-insensitive, normalized).
        If matched: return top 12 products from that category
        ordered by final_score (quality + freshness only,
        no text_score since we're not matching text).
        Return with flag: fallback: { level: 3, 
          inferred_category: '...' }

      LEVEL 4 — Trending products fallback:
        Return top 12 featured/trending products globally,
        ordered by (quality_score × 0.7 + freshness_score × 0.3).
        These are products where is_featured = true OR
        where they have the highest review counts in last 30 days.
        Return with flag: fallback: { level: 4,
          reason: 'no_match_found' }

      Rules for all fallback levels:
        - Never run the next level if the previous returned results.
        - Always include the fallback metadata in the API response.
        - Log every fallback activation to query_logs with a
          fallback_level column (add it if missing via migration).
        - The main search endpoint returns fallback results with
          HTTP 200, not 404.

  3b. Add fallback_level column to query_logs (if missing):
      Write an additive migration:
        ALTER TABLE query_logs
          ADD COLUMN IF NOT EXISTS fallback_level integer DEFAULT NULL;
        (NULL = no fallback needed, 1-4 = which level activated)
