IMPROVEMENT 3 — Mixed Language Query Handling
──────────────────────────────────────────────────────────────────

Goal: queries mixing Arabic and English (e.g. "iPhone رخيص",
"samsung هاتف", "laptop جديد") must work correctly.

  3a. Language detection per query token:
      Split the query into tokens. For each token, detect if it
      is Arabic script, Latin script, or numeric. Apply the
      appropriate normalization to each token independently.
      Arabic tokens → Arabic normalization pipeline
      Latin tokens → lowercase only
      Numeric tokens → kept as-is

  3b. Mixed query FTS construction:
      Build separate FTS sub-queries for Arabic tokens and English
      tokens, then combine with OR. This ensures PostgreSQL uses the
      correct text search configuration for each language.
      Example: "iPhone رخيص" →
        english_fts: to_tsquery('english', 'iphone')
        arabic_fts: to_tsquery('arabic', 'رخيص') -- or 'simple'
        combined: english_fts || arabic_fts

  3c. Cross-language synonym expansion for mixed queries:
      When a mixed query is detected, also expand using the
      cross-language synonyms (language: 'both') from the synonyms
      table. So "موبايل iPhone" also searches for "mobile iPhone
      هاتف جوال".

IMPROVEMENT 4 — Query Understanding Enhancements
──────────────────────────────────────────────────────────────────

  4a. Brand name protection:
      Build a brands list from existing product data:
        SELECT DISTINCT brand FROM products WHERE brand IS NOT NULL
      Cache this list in memory (refresh every 10 minutes).
      When a query contains a known brand name: apply an exact-match
      boost for that brand (multiply brand-matching products' text
      score by 1.5). This ensures "Samsung" always returns Samsung
      products first, not generic "phone" results.

  4b. Numeric query handling:
      If query contains a number:
        - 4-digit number (e.g. 2024) → likely a year, boost products
          with that year in title/description
        - Number with currency context (e.g. "بكم", "price", "سعر"
          near the number) → treat as price filter, add WHERE price
          BETWEEN (number * 0.8) AND (number * 1.2)
        - Pure numeric query (only digits) → search in product
          model numbers and SKUs in addition to title

  4c. Stop word refinement for Arabic:
      Add a Syrian-Arabic stop word list to the search pipeline.
      These words should be stripped from queries before FTS
      (they add noise, not signal):
        بدي, بدو, بدها, بدنا, بدكم, بدهم (Syrian "want")
        في, من, على, إلى, عن, مع, هاد, هاي, هدا, هاي
        شو, شوف, كيف, وين, ليش, امتى
        بس, لكن, يعني, هلق, هون, هناك
      CRITICAL: strip stop words ONLY when the remaining query has
      at least 1 non-stop-word token. If stripping would result in
      an empty query, keep the original query unchanged.

  4d. Query length handling:
      - Query longer than 100 characters: truncate to 100 chars,
        log a warning, continue processing.
      - Query with more than 8 tokens after stop word removal:
        take only the 8 highest-IDF tokens (most rare/specific).
        Implement IDF approximation using term frequency in
        query_logs — rarer search terms have higher IDF.
      - Single character query: return empty results immediately
        without hitting the DB. Add to i18n:
        search.error.queryTooShort: "اكتب كلمة كاملة للبحث" /
        "Type a full word to search"

    8. Handle numeric tokens
    9. Build FTS query (mixed language aware)
    10. Execute DB query with multi-signal ranking (Axis 4)
    11. Apply seller diversity post-processing
    12. Return results with intent + ranking metadata
  Document this pipeline as a comment block at the top of
  searchProcessor.ts.
