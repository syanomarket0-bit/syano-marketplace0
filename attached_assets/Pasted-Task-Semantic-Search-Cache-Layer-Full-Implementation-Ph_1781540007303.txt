STEP 1 — Python Embedding Service

Create a standalone Python microservice at:

This service has ONE job: convert text to vectors.
Nothing else. No business logic. No DB access.

  1a. Service stack:

      Install via: pip install fastapi uvicorn

                              https://download.pytorch.org/whl/cpu
      CPU-only torch (smaller, faster to install, sufficient for
      inference — no GPU needed on Railway/Replit).

  1b. Model to use:
      intfloat/multilingual-e5-small
      - 384-dimensional vectors
      - Supports Arabic + English natively
      - ~470MB download, cached after first run
      - Inference: ~10-30ms per query on CPU

      Model loading strategy:
        Load the model ONCE at startup using a global variable.
        Never reload per request.
        Log: "Model loaded in Xms" on startup.
        If model fails to load: log error, exit with code 1
        (do not serve requests with no model).

  1c. Service endpoints — implement ALL of these:

      GET /health
        Returns: { status: "ok", model: "multilingual-e5-small",
                   vector_dimensions: 384 }
        Used by Node.js to verify service is ready before search.

      POST /embed/query
        Body: { text: string, language?: "ar" | "en" | "mixed" }
        Returns: { embedding: number[], dimensions: 384,
                   processing_ms: number }
        Prefix strategy (CRITICAL for e5 model accuracy):
          Arabic or mixed: prepend "query: " to text before encoding
          English: prepend "query: " to text before encoding
          (e5 model requires "query: " prefix for queries and
           "passage: " prefix for documents — this is mandatory
           for correct similarity scores)
        Max text length: 512 tokens — truncate silently if longer.
        Processing: model.encode([prefixed_text],
                                  normalize_embeddings=True)
        Return embedding as plain Python list (JSON-serializable).

      POST /embed/batch
        Body: { texts: string[], type: "passage" | "query" }
        Returns: { embeddings: number[][], count: number,
                   processing_ms: number }
        Prefix strategy:
          type="passage": prepend "passage: " to each text
          type="query": prepend "query: " to each text
        Batch size limit: 64 texts per request maximum.
          If texts.length > 64: process in chunks of 64,
          concatenate results, return all.
        Processing: model.encode(prefixed_texts,
                                  batch_size=32,
                                  normalize_embeddings=True,
                                  show_progress_bar=False)
        This is used for bulk product embedding generation.

  1d. Service configuration:
      Port: 8001 (Node.js server stays on its current port)

STEP 4 — Semantic Search in searchProcessor.ts

  4a. Add embedding service client to searchProcessor.ts:
      Create a lightweight internal client (plain fetch — no new
      packages):

        async function getQueryEmbedding(
          query: string
        ): Promise<number[] | null> {
          try {
            const url = process.env.EMBEDDING_SERVICE_URL;
            if (!url) return null;
            const response = await fetch(`${url}/embed/query`, {
              method: 'POST',
              headers: { 'Content-Type': 'application/json' },
              body: JSON.stringify({ text: query }),
              signal: AbortSignal.timeout(2000) // 2s timeout
            });
            if (!response.ok) return null;
            const data = await response.json();
            return data.embedding ?? null;
          } catch {
            return null; // semantic search gracefully disabled
          }
        }

      CRITICAL: the 2-second timeout is non-negotiable.
      If the embedding service is slow or down, the search must
      fall back to FTS-only without error. Never block search
      for more than 2 seconds waiting for embeddings.

  4b. Add semantic search DB query function:
      Create a function semanticSearch that:
        Takes: embedding: number[], limit: number
        Builds this SQL query:
          SELECT
            p.id,
            p.name,
            p.name_ar,
            p.price,
            p.image_url,
            p.category,
            p.stock,
            1 - (p.embedding <=> $1::vector) AS semantic_score
          FROM products p
          WHERE
            p.embedding IS NOT NULL
            AND 1 - (p.embedding <=> $1::vector) > 0.45
            AND p.stock > 0
          ORDER BY semantic_score DESC
          LIMIT $2;

        The <=> operator is pgvector cosine distance.
        1 - distance = cosine similarity (0 to 1).
        Threshold 0.45 = minimum similarity to be considered
        relevant. Below this = noise, not a real match.
        Adjust column names to match actual schema.

        If pgvector is not available: catch the error, return
        empty array, log warning once (not on every query).

  4c. Hybrid scoring — combine FTS + Semantic:
      After getting both FTS results and semantic results,
      merge them using reciprocal rank fusion (RRF):

        function reciprocalRankFusion(
          ftsResults: SearchResult[],
          semanticResults: SearchResult[],
          k: number = 60
        ): SearchResult[] {
          const scores = new Map<number, number>();

          ftsResults.forEach((result, index) => {
            const current = scores.get(result.id) ?? 0;
            scores.set(result.id, current + 1 / (k + index + 1));
          });

          semanticResults.forEach((result, index) => {
            const current = scores.get(result.id) ?? 0;
            scores.set(result.id,
              current + 1 / (k + index + 1));
          });

          // Merge unique results, sort by RRF score
          const allResults = [
            ...ftsResults,
            ...semanticResults.filter(s =>
              !ftsResults.find(f => f.id === s.id))
          ];

          return allResults.sort((a, b) =>
            (scores.get(b.id) ?? 0) - (scores.get(a.id) ?? 0)
          );
        }

      Why RRF: it's the most robust fusion method that doesn't
      require score normalization across different ranking systems.
      Used by Elasticsearch, Pinecone, and Weaviate internally.

STEP 5 — LRU Cache Implementation
──────────────────────────────────────────────────────────────────

Create artifacts/server/src/services/searchCache.ts

Implement a production-grade in-memory LRU cache for search results.
No Redis. No external dependencies. Pure TypeScript.

  5a. Cache key strategy:
      The cache key must be a deterministic string that represents
      a unique search state. Build it from:
        key = sha256-like hash of:
          normalized_query + '|' +
          sort + '|' +
          category + '|' +
          minPrice + '|' +
          maxPrice + '|' +
          page + '|' +
          pageSize
      Use Node.js built-in crypto module:
        import { createHash } from 'crypto';
        const key = createHash('md5')
          .update(cacheKeyInput)
          .digest('hex');
      MD5 is fine for cache keys (not security — just uniqueness).
      Intent modifiers DO affect the cache key (they change results).

  5b. LRU implementation:
      Implement a proper LRU (Least Recently Used) cache using
      a Map + doubly-linked list pattern. Do NOT use a simple
      Map with size limit (that's not true LRU).

      Cache configuration:
        MAX_SIZE: 500 entries
          (500 unique queries * ~50KB avg = ~25MB max memory)
        TTL: 300 seconds (5 minutes) for normal queries
             60 seconds for intent=on_sale or intent=new_arrivals
             (sale/new content changes more frequently)
             600 seconds for fallback L4 (trending — changes slowly)
        TTL_BYPASS: never cache queries with debug=ranking
          (admins need fresh data)

      Cache entry structure:
        interface CacheEntry {
          key: string;
          value: SearchApiResponse;
          createdAt: number;    // Date.now()
          ttl: number;          // milliseconds
          hits: number;         // how many times served from cache
          queryText: string;    // for debugging
        }

  5c. Cache operations — implement all:

      get(key: string): SearchApiResponse | null
        1. Look up key in Map
        2. If not found: return null (cache miss)
        3. If found but expired (Date.now() > createdAt + ttl):
           delete from Map, return null (stale)
        4. Move entry to front of LRU list (most recently used)
        5. Increment hits counter
        6. Return value (cache hit)

      set(key: string, value: SearchApiResponse,
          ttl: number): void
        1. If key already exists: update value and move to front
        2. If cache is at MAX_SIZE: evict LRU entry (tail of list)
        3. Insert new entry at front of list
        4. Update Map

      invalidate(pattern?: string): void
        If pattern provided: delete all keys containing pattern
          (e.g., invalidate('electronics') clears electronics cache)
        If no pattern: clear entire cache
        Call this when: products are updated, new products added,
          prices change, reindex is run.

      getStats(): CacheStats
        Returns:
          {
            size: number,           // current entries
            maxSize: number,        // 500
            hitRate: number,        // hits / (hits + misses)
            totalHits: number,
            totalMisses: number,
            totalEvictions: number,
            memoryEstimateMB: number, // rough estimate
            oldestEntryAge: number  // seconds
          }

  5d. Cache integration in search.ts:
      Import the cache at the top of search.ts.
      In the main search handler:

        // Step 1: Check cache
        const cacheKey = buildCacheKey(params);
        const cached = searchCache.get(cacheKey);
        if (cached) {
          return res.json({
            ...cached,
            _cache: { hit: true, age: Math.floor(
              (Date.now() - cached._cacheCreatedAt) / 1000
            )}
          });
        }

        // ... run full search pipeline ...

        // Step 14: Write to cache
        const ttl = getTTLForResult(detectedIntent, fallbackLevel);
        searchCache.set(cacheKey, result, ttl);

      Add _cache field to SearchApiResponse type:
        _cache?: {
          hit: boolean;
          age?: number; // seconds since cached, only on hits
        }

  5e. Cache invalidation triggers:
      Invalidate the cache (full clear) when:
        - POST /api/admin/search/reindex is called
        - A product is created, updated, or deleted
        - A seller's status changes
      Add invalidateSearchCache() calls in the relevant route
      handlers. Find each handler and add the call — do not
      skip any.

  5f. Cache stats endpoint (admin only):
      GET /api/admin/search/cache-stats
        Returns the output of searchCache.getStats()
