CRITICAL CONTEXT — WHY THESE BUGS EXIST (read before diagnosing)
════════════════════════════════════════════════════════════════════

The search results page (/shop) has a sticky header composed of two
layers stacked vertically:

  Layer 1 (always present):
    The main site Navbar — height controlled by CSS var --navbar-height.

  Layer 2 (always present, sticky below Layer 1):
    The search bar div — contains:
      • Search input row (always visible)
      • "نتائج البحث عن ..." line (ONLY visible when debouncedQuery is
        truthy — this is the variable that controls the WITH-query state)
      • Tabs row: "المنتجات / المتاجر / الفئات"

Because Layer 2 contains a CONDITIONAL element (the results line), its
rendered height is VARIABLE:
  • WITHOUT query: shorter (input + tabs only)
  • WITH query:    taller  (input + results line + tabs)

Somewhere in the current code, the spacing below Layer 2 is set with a
STATIC value (a hardcoded rem or calc() using a fixed offset like 9rem or
9.5rem). This static value was accidentally calibrated to the WITH-query
height. Result:
  • WITH query:    static offset ≈ real header height → content starts
                  correctly below header ✓
  • WITHOUT query: static offset > real header height → content starts
                  too far down, BUT the sticky header is shorter, so
                  the top of the content is actually ABOVE where the
                  header ends → header covers the first product row ✗

This same mismatch causes the mobile filter panel top to be cut off
(Bug 2a) because the same static offset logic applies there.

Do NOT attempt another static value fix. The only correct solution is
dynamic measurement as specified in Bug 1 below.

BUG 1 — Desktop 1280px: first product row hidden under sticky header
         (no-query state only — /shop with no ?q= param)
════════════════════════════════════════════════════════════════════

Exact symptom confirmed in screenshot:
  At /shop (no query), the sticky tabs bar ("24 المنتجات / المتاجر /
  الفئات") overlaps and hides the top ~60-80px of the first product
  card row. Cards are present in the DOM but visually covered.
  At /shop?q=رخيص (with query) this does NOT occur — content is
  correctly positioned.

Step 1 — Locate the exact elements:
  In search/index.tsx, find:
    A) The sticky Layer 2 div — the one with `sticky` class and
       `style={{ top: "var(--navbar-height)" }}`. This is the element
       whose height changes between query/no-query states.
    B) The main content wrapper below it — the div that contains BOTH
       the desktop sidebar AND the product grid side by side. This is
       the element that currently has a static top offset (padding-top,
       margin-top, or a pt-* / mt-* Tailwind class) that needs to
       become dynamic.
    C) Any scroll-padding-top rule in index.css or inline style that
       uses a static calc() offset for this page.

Step 2 — Implement dynamic header height measurement:
  a) Add a React ref to element A (the sticky Layer 2 div):
       const searchHeaderRef = useRef<HTMLDivElement>(null);
       const [searchHeaderHeight, setSearchHeaderHeight] = useState(144);
       // 144px ≈ current static offset as initial fallback to avoid
       // layout flash before first measurement

  b) Add a useLayoutEffect that attaches a ResizeObserver to
     searchHeaderRef. On every resize callback, call:
       setSearchHeaderHeight(entry.contentRect.height);
     The observer must fire on:
       • Initial mount
       • Every time debouncedQuery changes (header grows/shrinks) —
         the ResizeObserver handles this automatically since it watches
         the element's actual size; no additional dependency needed
       • Window resize (ResizeObserver handles this too)
     Clean up the observer in the useLayoutEffect return function.

  c) Apply searchHeaderHeight dynamically to element B:
     Replace whatever static top-spacing currently exists on element B
     with:
       style={{ paddingTop: `${searchHeaderHeight}px` }}
     If element B currently uses a Tailwind pt-* class for this spacing,
     REMOVE that class and use the inline style instead — Tailwind static
     classes cannot be dynamic.

  d) Update scroll-padding-top (element C):
     If it is in index.css as a static rule, move it to an inline style
     on the <html> element or the page root div via useLayoutEffect:
       document.documentElement.style.scrollPaddingTop =
         `${searchHeaderHeight + 8}px`;
         // +8px = 0.5rem breathing room, same intent as current +0.5rem
     Clean this up on unmount. If it is already an inline style
     somewhere, replace the static calc() with the same formula.

  e) Attach the ref: add ref={searchHeaderRef} to element A's JSX.

Step 3 — Verify the fix logic:
  The fix is correct when:
    • On mount (no query): ResizeObserver fires, measures the short
      header height (e.g. ~88px), sets paddingTop = 88px on element B.
      First product row is fully visible.
    • After typing a query: header grows (e.g. to ~128px), ResizeObserver
      fires again, paddingTop updates to 128px. Content shifts down
      slightly. First product row remains fully visible.
    • On window resize: ResizeObserver fires, recalculates. Layout
      stays correct.

  same searchHeaderHeight state variable already created for Bug 1.

  logical filter section (category, price range, rating, checkboxes),
  wrap it in a div with consistent vertical spacing. Check how the

  values. If the desktop sidebar uses `space-y-4` between sections,
  the mobile panel must use `space-y-4` as well. If each section has
  a label with `text-sm font-medium mb-2`, the mobile version must
