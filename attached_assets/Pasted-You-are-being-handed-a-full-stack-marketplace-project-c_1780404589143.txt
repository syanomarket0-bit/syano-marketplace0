## 1. TECH STACK AT A GLANCE
| Layer | Technology |
|---|---|
| Monorepo | pnpm workspaces |
| Backend | Node.js + Express 5 + TypeScript |
| Database | PostgreSQL + Drizzle ORM (contract via schema files) |
| API Contract | OpenAPI 3.1 → Orval codegen → Zod + TanStack Query hooks |
| Frontend | React 19 + Vite 7 + Tailwind CSS v4 + shadcn/ui + wouter |
| Mobile | Expo (React Native) + expo-router |
| Auth | JWT Bearer tokens (7-day expiry), bcryptjs password hashing |
| i18n | i18next — English + Arabic (RTL support built-in) |
| Push Notifications | Web Push (VAPID) via service worker |
| Realtime | Server-Sent Events (SSE) for notifications stream |

## 3. ENVIRONMENT VARIABLES REQUIRED
### API Server (`artifacts/api-server`)

DATABASE_URL=postgresql://user:pass@host:5432/dbname # REQUIRED — Postgres connection string
SESSION_SECRET=<random-64-char-string> # REQUIRED — JWT signing secret
PORT=<number> # REQUIRED — server listen port
NODE_ENV=development|production
LOG_LEVEL=info # optional, default info
VAPID_PUBLIC_KEY=<base64> # optional — Web Push
VAPID_PRIVATE_KEY=<base64> # optional — Web Push

### Marketplace Web (`artifacts/marketplace`)

PORT=<number> # REQUIRED — Vite dev server port
BASE_PATH=/ # REQUIRED — URL base path (e.g. "/" or "/marketplace")

EXPO_PUBLIC_DOMAIN=<replit-dev-domain-or-prod-domain> # e.g. "abc123.janeway.replit.dev"

The mobile app constructs the API base URL as `https://${EXPO_PUBLIC_DOMAIN}`.

## 4. HOW TO RUN (from repo root)
```bash

pnpm install
# 2. Push the database schema (creates/updates all tables)

# 3. Seed the first admin user (run once)

All tables are in lib/db/src/schema/. The client is exported from @workspace/db.

users
Column	Type	Notes
id	serial PK

phone	text UNIQUE	nullable
password_hash	text NOT NULL	bcryptjs

verified_at	timestamp	null = unverified

products
Column	Type	Notes
id	serial PK	
seller_id	int FK→users	
name	text NOT NULL	English name

Indexes: category, seller_id, featured, created_at, stock

product_variant_groups / product_variant_options / product_variants / product_variant_values
Four-table variant system: a product has variant groups (e.g. "Color"), each group has options (e.g. "Red", "Blue"), combined into specific variants with their own price/stock.

cart_items
Column	Type
id	serial PK
user_id	int FK→users

variant_id	int FK→product_variants nullable

orders
Column	Type	Notes
id	serial PK	
customer_id	int FK→users	
total	numeric(10,2)	
status	enum	pending→processing→shipped→delivered|cancelled|refunded
shipping_address	text	full address string
customer_phone	text	
city	text	
delivery_notes	text	
tracking_number	text	
shipping_company	text	
shipped_at	timestamp	
created_at / updated_at	timestamp	
Important business rule: Stock is only decremented when order status reaches delivered (stock-on-delivery pattern), not at order creation.

order_items
Column	Type
id	serial PK
order_id	int FK→orders
product_id	int (no FK — intentional, product may be deleted)

seller_id	int

Tracks every status transition with who changed it (changedBy, changedByRole) and optional notes.

reviews
Product reviews: id, product_id, user_id, rating (1-5), comment, created_at.
One review per user per product enforced at API level.

seller_reviews
Seller-specific ratings: communication_rating, shipping_rating, professionalism_rating (all 1-5), comment.
Unique per (seller_id, customer_id) pair.

seller_applications
Full onboarding form: store_name, phone, city, address, category, categories[], description, social_links, website, id_image_url, store_logo, store_banner, store_slug, status (pending|approved|rejected), admin_notes, rejection_reason, reviewed_at, reviewed_by_id.

conversations + messages
Buyer↔Seller messaging:

conversations: customer_id, seller_id, product_id (optional context), status (active|archived), last_message_at
messages: conversation_id, sender_id, body, read_at, flagged (boolean for moderation)
store_follows
Many-to-many: follower_id → seller_id. Unique per pair.

notifications
System notifications: user_id, type, title, body, data (JSON), read_at, created_at.

push_subscriptions
Web Push: user_id, endpoint, p256dh, auth (VAPID keys per subscription).

platform_settings
Key-value store for admin-controlled site settings (e.g. maintenance mode, featured category).

audit_log
Admin action log: admin_id, action, target_type, target_id, details (JSON), created_at.

10. KEY BUSINESS LOGIC & DECISIONS
User Roles & Flow
Register → role: "customer" (default)
    └─► Apply to become seller (POST /seller-applications)
        └─► Admin approves → role flips to "seller"
            └─► Seller can create products, manage orders, view dashboard

Order Lifecycle
pending → processing → shipped → delivered
                              ↘ cancelled
                              ↘ refunded

Stock decrements only at delivered, not at order creation (stock-on-delivery pattern)
Every transition is logged in order_status_history
Seller sees orders containing their seller_id in order_items
Search System
products.search_tokens column stores pre-computed trigram-friendly text (name + name_ar + category + description)
Backend uses pg_trgm extension for fuzzy matching
lib/search-startup.ts ensures the extension is enabled on first boot
Frontend multilingual: Arabic query → also searches English equivalents via synonym expansion
Notifications (SSE Fan-out)
Route: GET /api/notifications/stream — long-lived SSE connection per authenticated client
Fan-out function in lib/fanout.ts — pushes to all open SSE connections for a user
Triggers: new order (seller notified), order status change (customer notified), stock low (seller notified)
Designed to be replaced with a proper queue (Bull/Redis) by swapping fanout.ts
Seller Store Pages
URL pattern: /store/:slug — slug comes from seller_applications.store_slug
Store page shows: seller info + trust badges + products + follower count + seller reviews
Trust level (users.trust_level) is computed from: verified email + order completion rate + account age
Push Notifications (Web)
Service worker at public/sw.js handles Web Push events
VAPID keys required (VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY, VAPID_EMAIL env vars)
In development/preview: service worker is intentionally NOT registered (prevents stale cache). Only active in production builds.

12. COMMON PITFALLS
Never edit lib/api-zod/ or lib/api-client-react/ directly — they are 100% generated. Edit openapi.yaml and run codegen.
DATABASE_URL must be set before running the API server or any db command — it throws a hard error otherwise.
SESSION_SECRET should be a strong random string in production. Default "dev-secret-change-in-prod" is only for local dev.
pnpm only — the root package.json has a preinstall script that blocks npm/yarn.
EXPO_PUBLIC_DOMAIN must be set for mobile to reach the API — it's the publicly accessible domain (not localhost).
Schema changes require both a db push AND potentially a codegen run if you changed API shapes.
Tailwind v4 uses @import "tailwindcss" in index.css — there is NO tailwind.config.js. All customization is via CSS custom properties in :root.
